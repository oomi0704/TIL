# TIL - 2026-06-04 AWS SAP 学習メモ

---

## 1. API Gateway の配置

- API Gateway はVPC外のマネージドサービス
- **3種類のエンドポイント**
  - エッジ最適化：CloudFront経由、パブリック
  - リージョン：同一リージョン向け、パブリック
  - プライベート：VPCエンドポイント（Interface型）経由でのみアクセス可
- プライベートAPIへオンプレからアクセスするには **Direct Connect or VPN + VPCエンドポイント** が必要

---

## 2. EDR / EPP

- **EPP**（Endpoint Protection Platform）：既知の脅威を事前に防ぐ（侵入させない）
- **EDR**（Endpoint Detection and Response）：侵入後に検出・調査・対応する
- AWSでの対応：GuardDuty（EDR的）／Inspector（EPP的）／Security Hub（統合管理）

---

## 3. AWS Application Discovery Agent

- オンプレミスサーバーにインストールするエージェント
- 収集情報：CPU・メモリ・プロセス・サーバー間の依存関係・パフォーマンスデータ
- **Agentあり vs Agentなし（Agentless）**
  - Agentあり：物理・仮想・VM対応、依存関係マップ取得可
  - Agentなし：VMware環境のみ、基本情報のみ

---

## 4. AWS Migration Hub

- 複数の移行ツールの進捗を**一元管理するダッシュボード**
- 移行作業自体は行わない
- **ホームリージョンは1つだけ**設定可能、後から変更不可
- 連携サービス：Application Discovery / MGN / DMS

---

## 5. Migration Evaluator

- 移行前の**コスト試算・ビジネスケース作成**ツール
- 「AWSに移行したらいくら節約できるか」を分析
- Application Discovery との違い：
  - Evaluator → コスト試算（経営層向け）
  - Discovery → 技術情報収集・依存関係把握

---

## 6. AWS Compute Optimizer

- 機械学習で**リソースの最適サイズを推奨**
- 対応リソース：EC2・Auto Scaling・EBS・Lambda・ECS（Fargate）
- 推奨分類：Over-provisioned / Under-provisioned / Optimized
- デフォルトで**無効**、有効化後14日間でデータ収集
- Cost Explorerとの違い：OptimizerはリソースサイズCost Explorerはコスト可視化

---

## 7. Storage Gateway 3種類

| 種類 | プロトコル | 保存先 | 用途 |
|------|-----------|--------|------|
| ファイルゲートウェイ | NFS / SMB | S3 | ファイル共有 |
| ボリュームゲートウェイ | iSCSI | S3 / EBS | ブロックストレージ |
| テープゲートウェイ | iSCSI（VTL） | S3 / Glacier | バックアップ |

### ボリュームゲートウェイの2モード

- **保管型（Stored）**：データ本体はローカル、S3はスナップショット（バックアップ）
- **キャッシュ型（Cached）**：データ本体はS3、ローカルはよく使うデータのキャッシュのみ

> 通常のキャッシュとは主従関係が逆！S3が主役でローカルが従

## 8. AWS Systems Manager vs Application Discovery

| | Systems Manager | Application Discovery |
|--|----------------|----------------------|
| 目的 | 稼働中サーバーの運用・管理 | 移行前の情報収集 |
| タイミング | 移行後・日常運用 | 移行前 |
| 主な機能 | Session Manager / Patch Manager / Run Command / Parameter Store | サーバー情報収集・依存関係マップ |

## 9. OWASP と AWS WAF

- **OWASP**：Webアプリセキュリティの非営利団体
- **OWASP Top10**：重大なセキュリティリスクTop10（SQLi・XSS・SSRFなど）
- AWS WAFにはOWASP対応の**マネージドルール**が用意されている
- WAFはサービスに「紐付ける」形で配置
  - CloudFront用WAFは **us-east-1 で作成必須**

---

## 10. AWS WAF の配置と前段・後段の概念

```
ユーザー（インターネット）
　↓
【前段】Shield → CloudFront → WAF
　↓
【後段】ALB → EC2（VPC内）
```

- **前段** = ユーザーのリクエストが最初に通る場所
- **後段** = 前段を通過した後に到達する場所
- 多層防御 = CloudFront+WAF → ALB+WAF の二段構え

---

## 11. GuardDuty vs Inspector

| | GuardDuty | Inspector |
|--|-----------|-----------|
| 目的 | 不審な動きをリアルタイム検知 | 脆弱性・パッチ不足を発見 |
| イメージ | 警備員（監視カメラ） | 建物の点検業者 |
| タイミング | リアルタイム監視 | 定期的なスキャン |
| 分析対象 | CloudTrail・VPCフローログ・DNSログ | EC2・ECR・Lambda |

- **GuardDuty** → 攻撃中・攻撃後の検知
- **Inspector** → 攻撃される前の予防（脆弱性修正）
