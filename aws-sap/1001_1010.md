# AWS SAP 学習まとめ（問題1002〜1010）

---

## 問題1002：vCPU

### vCPUとは
- 物理CPUコアを仮想化したもの
- `物理CPUコア × スレッド数 = vCPU数`
- EC2インスタンスに割り当てられる仮想的なCPUの数

### SAPでのポイント
- アカウントごとにリージョンあたりのvCPUクォータ（上限）がある
- 大量のEC2を起動する際は **Service Quotas** で上限緩和申請が必要
- スポット・オンデマンド・リザーブドでそれぞれ別カウント

---

## 問題1002（補足）：ワークロードとは
- システムが処理する仕事・作業の総称
- 例：Webアプリ処理、バッチ処理、機械学習処理など
- ワークロードの特性（常時稼働 / 突発的 / 予測可能）によってリザーブド・スポット・オンデマンドの使い分けが問われる

## 問題1002（補足）：AWS Compute Optimizer
- リソースの使用状況を機械学習で分析し、最適なリソースを推奨するサービス
- 対象：EC2・Auto Scalingグループ・EBSボリューム・Lambda・ECS（Fargate）
- CloudWatchの過去14〜93日分のメトリクスを分析
- デフォルトでOFF（有効化が必要）
- Organizations全体に一括適用可能

### CloudWatchメトリクス
| メトリクス | 取得 |
|---|---|
| CPUUtilization | デフォルトで取得可能 |
| NetworkIn/Out | デフォルトで取得可能 |
| DiskReadOps/WriteOps | デフォルトで取得可能 |
| メモリ使用率 | **CloudWatch Agentが必要** |

## 問題1003：インターコネクト（Direct Connect）

### インターコネクトとは
- AWS Direct ConnectにおけるAWSとネットワークプロバイダー間の専用物理接続
- AWSパートナー（通信事業者）がインターコネクトを持ち、複数顧客に分けて提供

### 構成イメージ
```
企業A ┐
企業B ├→ パートナー →[インターコネクト]→ AWS
企業C ┘
```

### ホスト型接続
- パートナーがインターコネクト経由で顧客に提供する接続
- 50Mbpsなど小さい帯域から利用可能（コスト削減に有効）

## 問題1004：データベース移行関連

### JDBC接続
- JavaアプリケーションからDBにアクセスするためのAPI
- RDS Proxy：Lambdaなどからの接続数爆発を防ぐ
- Secrets Manager：JDBC接続情報（パスワード）の安全な管理・自動ローテーション

### AWS SCT（Schema Conversion Tool）
- 異なるDBエンジン間でスキーマを変換するツール
- SCT（スキーマ変換）+ DMS（データ移行）がセット
- 同種DB間の移行はSCT不要、DMSのみでOK

| 移行元 | 移行先 |
|---|---|
| Oracle | Aurora PostgreSQL / MySQL |
| SQL Server | Aurora / RDS |
| Teradata / Oracle DW | Redshift |

### Foreign Data Wrapper（FDW）
- PostgreSQLから外部DBを自分のテーブルのように参照できる仕組み
- SCT・DMS（移行）と違い、**移行せず外部DBを参照し続ける**用途

### PostGIS
- PostgreSQLに地理空間データを扱う機能を追加する拡張機能
- Aurora PostgreSQL / RDS PostgreSQLで利用可能
- Oracle Spatial → PostGISへの移行はSCTで対応可能

### Oracle → DynamoDB移行は不適切
- OracleはRDB（JOIN・トランザクション多用）
- DynamoDBはNoSQL（JOINできない）
- SCTもRDB→NoSQL変換は非対応
- Oracle移行の正解は **Aurora / RDS**

## 問題1005：AWS Systems Manager

### 主な機能
| 機能 | 内容 |
|---|---|
| Session Manager | SSHなしでEC2にブラウザからアクセス |
| Parameter Store | 設定値・パスワードを安全に保存 |
| Patch Manager | OSパッチを自動適用 |
| Run Command | 複数サーバーに一括コマンド実行 |
| Inventory | インストール済みソフトの一覧取得 |
| Automation | 運用作業の自動化 |

### Parameter Store vs Secrets Manager
| | Parameter Store | Secrets Manager |
|---|---|---|
| コスト | 安い | 高め |
| 自動ローテーション | ❌ | ✅ |
| 用途 | 設定値管理 | パスワード管理 |

## 問題1007：Apache Tomcat

### Tomcatとは
- JavaのWebアプリ（Servlet・JSP）を実行するアプリケーションサーバー

### AWSへの移行先
| 移行先 | 特徴 |
|---|---|
| **Elastic Beanstalk** | Tomcat公式サポート・最も手軽 |
| **ECS / Fargate** | コンテナ化して移行 |
| **EC2** | そのまま移行（リフト＆シフト） |

### VPCエンドポイント vs VPCピアリング
| | VPCエンドポイント | VPCピアリング |
|---|---|---|
| 用途 | VPC→AWSサービスへのプライベート接続 | VPC同士の接続 |
| リージョン越え | ❌ | ✅ |
| 種類 | ゲートウェイ型（無料）/ インターフェイス型（有料） | - |

- VPCが多い場合は **Transit Gateway** が正解

## 問題1008：ネットワーク接続

### Site-to-Site VPN
- オンプレ ↔ AWS VPCをインターネット経由で暗号化接続
- 構成要素：カスタマーゲートウェイ（オンプレ側）+ 仮想プライベートゲートウェイ（AWS側）
- VPNトンネルは**必ず2本**（冗長構成）

### Direct Connect vs Site-to-Site VPN
| | VPN | Direct Connect |
|---|---|---|
| 回線 | インターネット | 専用線 |
| コスト | 安い | 高い |
| 安定性 | 低め | 高い |
| 導入速度 | 速い（即日） | 遅い（数週間〜） |

### リージョン間接続
| サービス | リージョン越え |
|---|---|
| Transit Gatewayピアリング | ✅ 多数VPC向け |
| VPCピアリング | ✅ シンプルな2VPC向け |
| Site-to-Site VPN | ❌ |
| VPCエンドポイント | ❌ |

### 共有サービスアカウント
- 複数アカウントから共通利用するサービスをまとめた専用アカウント
- DNS・AD認証・監視・セキュリティなどを一元提供
- Organizations（ガバナンス・統制）とは役割が異なる
- Transit Gatewayのハブ＆スポーク構成で接続するのが定番

## 問題1009：TTL（DNS文脈）

### TTLとは
- Time To Live：データの有効期限・生存時間

### DNSのTTL
- DNSレコードのキャッシュ有効期間
- TTL長い → 問い合わせ少ない・変更が反映されにくい
- TTL短い → 変更がすぐ反映される・問い合わせ増

### ElastiCacheとTTLの関係
- フェイルオーバー時にTTLが長いと古いIPを使い続けて接続エラーになる
- 解決策：アプリ側のTTLを短く設定する

### サービスごとのTTL
| サービス | TTLの用途 |
|---|---|
| Route 53 | DNSレコードのキャッシュ時間 |
| DynamoDB | レコードの自動削除 |
| CloudFront | キャッシュの有効期間 |
| ElastiCache | キャッシュの有効期間 |

### ワークロードの種類と適したサービス
| ワークロード | 特徴 | 適したサービス |
