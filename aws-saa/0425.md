## TIL: AWS SAA 用語整理

### EC2 の購入オプション

| 用語 | 使いどころ | 覚え方 |
|---|---|---|
| オンデマンドインスタンス | 短期・不規則・検証用途 | 使った分だけ払う。最も柔軟だが割引は少ない |
| リザーブドインスタンス | 長期間ずっと使うEC2 | 1年/3年の利用予約で割引 |
| スケジュールリザーブドインスタンス | 毎週・毎月など決まった時間だけ使う | 定期スケジュールの容量予約。現在は新規購入不可 |
| Savings Plans | 柔軟に割引を受けたい | 1時間あたりの利用料金をコミットして割引 |

- **不規則・短期** → オンデマンド
- **常時使うEC2が決まっている** → リザーブドインスタンス
- **決まった曜日・時間だけ使う** → スケジュールリザーブドインスタンス
- **柔軟に割引を受けたい** → Savings Plans

### VPN と NAT

| 用語 | 役割 |
|---|---|
| VPN | ネットワーク同士を安全に接続する暗号化トンネル |
| NAT | プライベートIPをパブリックIPに変換して外向き通信させる |

- **オンプレミスとVPCを暗号化接続** → Site-to-Site VPN
- **プライベートサブネットのEC2からインターネットへ出る** → NAT Gateway
- VPNは**接続・暗号化**、NATは**IP変換・外向き通信**。

### ActiveMQ と SQS

| 用語 | 特徴 |
|---|---|
| ActiveMQ / Amazon MQ | JMS、AMQP、MQTT、STOMPなど標準プロトコル互換のメッセージブローカー |
| SQS | AWSフルマネージドのメッセージキュー |

- **既存アプリがJMS/ActiveMQ互換を要求** → Amazon MQ
- **AWSネイティブに非同期キューを作る** → SQS
- **サーバー管理を避けたい・高スケール** → SQS

### VPCフローログ

VPC内のネットワーク通信メタデータを記録する機能。

- 対象: VPC、サブネット、ENI
- 記録内容: 送信元IP、宛先IP、ポート、プロトコル、通信量、許可/拒否など
- 出力先: CloudWatch Logs、S3、Kinesis Data Firehose
- **通信の中身は見えない**。HTTP本文、URL、SQL文などは記録しない。

試験では、**通信が許可/拒否されたか調査したい**、**不審なIPとの通信を監査したい**ならVPCフローログ。

### Amazon Macie と GuardDuty

| 用語 | 見るもの | 得意なこと |
|---|---|---|
| Amazon Macie | 主にS3内のデータ | 個人情報、クレカ番号、認証情報などの検出 |
| Amazon GuardDuty | CloudTrail、VPC Flow Logs、DNSログなど | 不審なAPI操作、侵害疑い、異常通信の検出 |

- **S3内の機密データを見つけたい** → Macie
- **攻撃・不審な挙動・侵害の兆候を検出したい** → GuardDuty
- Macieは**データの中身**、GuardDutyは**怪しい動き**を見る。

### データ系サービス

| サービス | 一言でいうと |
|---|---|
| AWS Transfer Family | SFTP/FTPS/FTPでS3やEFSへファイル転送 |
| Amazon DataZone | 組織内データを探す・共有する・管理するデータポータル |
| AWS Data Exchange | 外部データセットを取得・購読するマーケットプレイス |
| AWS Glue | ETL。データを抽出・変換・ロードする |
| Glue Data Catalog | テーブル定義やスキーマなどメタデータを管理する |

- **SFTP/FTP連携** → Transfer Family
- **社内データの検索・共有・承認管理** → DataZone
- **社外データの購読** → Data Exchange
- **データ加工・ETL** → Glue
- **テーブル定義・スキーマ管理** → Glue Data Catalog

### Route 53 と Route 53 Record

| 用語 | 意味 |
|---|---|
| Amazon Route 53 | AWSのDNS・ドメイン管理サービス |
| Hosted Zone | ドメインのDNS設定を入れる場所 |
| Record | 名前解決の具体的な設定 |

- **Route 53** = DNSサービス全体
- **Hosted Zone** = `example.com` 用の住所録
- **Record** = `www.example.com -> ALB` のような1件の設定
- **Alias Record** = ALB、CloudFront、S3静的Webサイト、API GatewayなどAWSリソースに向けるRoute 53独自機能

### AppSync と API Gateway

| 用語 | 得意なAPI |
|---|---|
| AWS AppSync | GraphQL API |
| Amazon API Gateway | REST API、HTTP API、WebSocket API |

- **GraphQL APIを作る** → AppSync
- **REST/HTTP APIを作る** → API Gateway
- **Lambdaを外部公開するAPIにする** → API Gateway
- **リアルタイムGraphQL Subscription** → AppSync
- **WebSocket API** → API Gateway

AppSyncは**GraphQL特化**、API Gatewayは**汎用APIの入口**。

### Route 53 フェイルオーバーと RDS フェイルオーバー

| 項目 | Route 53 フェイルオーバー | RDS フェイルオーバー |
|---|---|---|
| 切り替える対象 | DNSの名前解決先 | DBのプライマリ |
| 主な用途 | Web/API/サイトのDR | DBの高可用性 |
| 判定 | ヘルスチェック | RDSが障害検知 |
| 範囲 | AZ間もリージョン間も可能 | 主にMulti-AZでAZ間 |
| データ同期 | Route 53自体はしない | RDS側で同期/レプリケーション |

- **入口の切り替え** → Route 53
- **DB本体の切り替え** → RDS Multi-AZ
- **リージョン間DR** → Route 53 + 別リージョン構成
- **同一リージョン内のDB高可用性** → RDS Multi-AZ

### Migration Hub Orchestrator と Application Migration Service

| サービス | 役割 |
|---|---|
| AWS Application Migration Service | サーバーをAWSへ実際に移行する |
| AWS Migration Hub Orchestrator | 移行作業の手順・テンプレート・進捗を管理する |

- **オンプレサーバーを最小停止時間でEC2へ移行** → Application Migration Service
- **移行手順をテンプレートで管理・自動化** → Migration Hub Orchestrator
- Application Migration Serviceは**移す道具**、Orchestratorは**移行手順の指揮役**。

### スティッキーセッション、RDSエンドポイント、非同期並列実行

| 用語 | 何の話か |
|---|---|
| スティッキーセッション | ALB/ELBで同じユーザーを同じEC2へ振り分ける |
| RDSエンドポイント | アプリがDBへ接続するためのDNS名 |
| 非同期並列実行 | キューやワークフローで処理を分散・並列化する設計 |

- **同じユーザーを同じEC2へ** → スティッキーセッション
- **DBの接続先を安定させる** → RDSエンドポイント
- **RDS Multi-AZフェイルオーバー後も同じ接続先名を使う** → RDSエンドポイント
- **処理を待たせず並列化** → SQS + Lambda、Step Functions、EventBridge、ECS、AWS Batch
- セッション情報はEC2ローカルではなく、ElastiCacheやDynamoDBに外部化する設計がよく正解になる。

### DynamoDB Streams、GSI、DAX、Events

| 用語 | 目的 |
|---|---|
| DynamoDB Streams | データ変更履歴をストリームとして記録する |
| GSI | 主キー以外の属性で効率よく検索する |
| DAX | DynamoDB専用インメモリキャッシュで読み取りを高速化する |
| DynamoDB Events | StreamsをイベントとしてLambdaなどで処理する |

- **データ変更を検知したい** → DynamoDB Streams
- **変更をきっかけにLambdaを動かしたい** → DynamoDB Streams + Lambda
- **主キー以外で検索したい** → GSI
- **読み取りを高速化したい** → DAX
- DAXは読み取り高速化向き。強い整合性が必要な読み取りには向きにくい。

### S3 Glacier Instant Retrieval と Flexible Retrieval

| ストレージクラス | 取り出し速度 | 用途 |
|---|---|---|
| S3 Glacier Instant Retrieval | ミリ秒 | まれに使うが、必要時はすぐ取り出したい |
| S3 Glacier Flexible Retrieval | 数分〜数時間 | 取り出しを待てるアーカイブ |
| S3 Glacier Deep Archive | 12〜48時間 | 最安クラスの長期保管 |

- **めったに使わないが即時アクセスが必要** → Glacier Instant Retrieval
- **数分〜数時間待てるアーカイブ** → Glacier Flexible Retrieval
- **半日以上待てる長期保管** → Glacier Deep Archive

### ECS と EKS

| 用語 | 特徴 |
|---|---|
| ECS | AWS独自のコンテナオーケストレーション |
| EKS | KubernetesをAWSでマネージド利用する |
| Fargate | コンテナをサーバーレスで実行する起動タイプ |

- **AWSでシンプルにコンテナ運用** → ECS
- **Kubernetes標準・オープンソースツール・移植性重視** → EKS
- **Helm、Argo CD、Prometheus、Grafana、Istioなどを活用** → EKS
- **EC2のプロビジョニングや管理を避けたい** → ECS on Fargate
- **Kubernetes要件がない、運用を簡単にしたい** → ECS + Fargate

### AWS Service Catalog

AWS Service Catalogは、管理者が承認したAWS構成を、利用者がセルフサービスで安全に作れるようにするサービス。

- 中身はCloudFormationテンプレートなど
- 承認済みEC2、RDS、VPC構成などを製品として提供できる
- タグ、IAM制約、標準構成を強制しやすい
- ガバナンスを効かせたセルフサービスに使う

開発環境と本番環境の使い分けにも使えるが、**環境分離そのものの主役は複数AWSアカウント + AWS Organizations**。

- **環境ごとの標準構成を配布** → Service Catalog
- **開発/本番を強く分離** → 複数AWSアカウント + Organizations
- **権限やガードレール** → IAM / SCP
