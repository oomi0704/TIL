## IAM Identity Center の SAML フェデレーション

- **外部の SAML 2.0 IdP**（Entra ID、Okta、Google Workspace など）と **IAM Identity Center** を信頼関係でつなぎ、**企業アカウントで SSO して AWS／連携アプリに入る**仕組み。
- **IdP**：認証の正本。ユーザー名／パスワード・MFA はここ。
- **SP（サービスプロバイダ）**：Identity Center。SAML アサーションを検証し、アカウント／アプリへのアクセスを許可。

### 試験向けメモ

- **人間向けアクセスの集約**、**複数 AWS アカウント**への SSO の文脈で出やすい。
- **従来の IAM SAML（AssumeRoleWithSAML）**はロール信頼で STS に直接入る話。**Identity Center**はハブ型の現行パターンとして区別する。

## Redshift（SAA で押さえておきたい論点）

- **列指向・MPP のマネージド DWH**。**OLAP・大規模集計**向き。**OLTP の主置き場**としては不適切、という対比が典型。
- **Redshift Spectrum**：S3 をロードせず SQL で参照。**データレイク＋DWH**のハイブリッド。
- **取り込み**：`COPY`（S3）、Kinesis Data Firehose、DMS、Glue 連携など。
- **DR／可用性**：スナップショット、別リージョンコピー、Multi-AZ（RA3 文脈）。
- **セキュリティ**：VPC、Enhanced VPC Routing（VPC 内経路）、KMS、TLS。
- **負荷・コスト**：Concurrency Scaling（バースト）、RA3＋マネージドストレージ、RI／Savings Plans（安定負荷）、**Redshift Serverless**（変動大・運用最小）。
- **他サービスとの棲み分け**：RDS／Aurora＝OLTP、Athena＝S3 上サーバレス SQL、EMR＝Spark 等の柔軟な分散処理。

### 判定の軸

データ規模・**クエリが分析かトランザクションか**・**クラスタ管理の要否**で Redshift／Athena／EMR／RDS を切る。

## キャッシュとクッキーの違い

| | **キャッシュ** | **クッキー** |
|---|----------------|--------------|
| 主な目的 | **表示を速くする**（同じリソースの再取得を減らす） | **サーバがユーザーを識別し状態を保持**する |
| 典型の中身 | HTML・CSS・JS・画像などの**コピー** | セッションID、言語設定など**小さなテキスト** |
| リクエストで送る？ | 基本は送らない（条件付き再取得など） | **同じサイトへのリクエストに `Cookie` で送る** |

- **キャッシュ**：`Cache-Control`／`ETag` など HTTP で制御。CDN・ブラウザで同じ考え方。
- **クッキー**：`Set-Cookie` で保存、`HttpOnly`／`Secure`／`SameSite` で安全性を調整。

### 覚え方

**キャッシュ＝速くするためのファイルのコピー**。**クッキー＝サーバとの会話用の小さなメモ（毎回同梱）**。
