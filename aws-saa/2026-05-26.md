# TIL - AWS SAA 学習まとめ (2026-05-26)

---

## S3 ストレージクラス

- **S3 Standard-IA** → アクセスが少ないとわかっているデータ向け・取り出し料金あり・最低30日保存
- **S3 Intelligent-Tiering** → アクセスパターンが予測できないデータ向け・自動で階層移動・取り出し料金なし
- **S3 Express One Zone** → 単一AZ・超低レイテンシ・ML/HPC向け（試験優先度は低め）

---

## クロスアカウントアクセス

- **両方向の許可が必要**：アカウントB（リソース側）のバケットポリシー ＋ アカウントA（利用者側）のIAMポリシー
- ベストプラクティスは **IAMロール ＋ AssumeRole**
- 方法：バケットポリシー / IAMロール / ACL（非推奨）

---

## DR戦略（コスト・RTO順）

| 戦略 | RTO | コスト |
|---|---|---|
| Backup & Restore | 時間〜日 | 最安 |
| Pilot Light | 数十分〜時間 | 安い |
| Warm Standby | 数分〜十数分 | やや高い |
| Multi-Site Active/Active | ほぼゼロ | 最高 |

- **Pilot Light** = DBのみ起動済み・APPは障害時に起動
- **Warm Standby** = 縮小構成で常時稼働・障害時にスケールアップ

---

## スティッキーセッション

- 同じユーザーのリクエストを**常に同じEC2に送り続ける**仕組み
- CookieでEC2と紐付け（ELB生成 or アプリ生成）
- 欠点：特定EC2に負荷が偏る
- 代替案：**ElastiCache（Redis）** でセッションを共有管理

---

## VPCピアリング

- 同一リージョン・異なるリージョン・異なるアカウント **すべて接続可能**
- **推移的ルーティング不可**（A↔B↔CでAとCは直接ピアリング必要）
- 多数のVPC接続には **Transit Gateway** が正解

### クロスリージョンの制限
- IPv6通信不可
- DNSホスト名解決不可（IPアドレス直接指定が必要）
- セキュリティグループ参照不可（CIDRで指定）
- VPC CIDRは固定なのでCIDR指定で問題なし

---

## IAM

- **IAMポリシー** = 「何を許可・拒否するか」のルール文書
- **IAMロール** = ポリシーをまとめた入れ物（AWSサービスや他アカウントに付与）
- **ロール = ポリシーの集合体**
- ロール固有の特徴：一時的な認証情報を発行（AssumeRole）・アクセスキー不要

---

## Organizations / Control Tower

- **AWS Organizations** = 複数アカウントの組織管理
- **OU** = 組織単位でアカウントをグループ化
- **SCP** = OUやアカウントが持てる権限の上限設定（IAMポリシーとの両方許可が必要）
- **Control Tower** = マルチアカウントのベスト構成を自動構築・ガードレールで管理

## VPCエンドポイント

| 種類 | 対応サービス | 料金 |
|---|---|---|
| ゲートウェイ型 | **S3・DynamoDBのみ** | 無料 |
| インターフェース型 | SQS・SNS・Lambda等多数 | 有料 |

- **ゲートウェイ型 = S3とDynamoDBだけ** を必ず覚える

## Global Accelerator

- 配置順：**Global Accelerator → WAF → CloudFront → ALB → バックエンド**
- TCP・UDP両対応
- UDPの振り分けは **NLB**（ALBはUDP非対応）
- 固定IPアドレス（Anycast IP）

### CloudFrontとの違い
| | Global Accelerator | CloudFront |
|---|---|---|
| 用途 | TCP/UDP通信の高速化 | コンテンツのキャッシュ配信 |
| IP | 固定 | 変動 |

## リンクローカルアドレス

- **169.254.169.254** = EC2インスタンスメタデータの取得に使用
- ルーターを越えられないため外部から悪用されない

## ACM・Route 53・API Gateway

- ACMが接続できる：**CloudFront・ALB・API Gateway**
- Route 53はACMの **DNS検証** に使う（直接接続ではない）
- CloudFrontでACMを使う場合は **バージニア北部リージョン必須**

---

## ELB・ALB・NLB

- **ALB** = HTTP/HTTPS（L7）・UDP非対応
- **NLB** = TCP/UDP（L4）・UDP対応
- ALBの後ろにDBは直接繋がらない（アプリ層を経由する）
- **RDS Proxy** = アプリとDBの間に入って接続を効率化

## DB関連

### トラフィック自動制御の使い分け
| やりたいこと | 正解 |
|---|---|
| 接続数過多（Lambda等） | RDS Proxy |
| 読み取り負荷が高い | Auroraリードレプリカ＋オートスケーリング |
| 予測不能なトラフィック | Aurora Serverless |
| 障害時の自動切り替え | Route 53フェイルオーバー |

### Aurora Serverless v1 vs v2
| | v1 | v2 |
|---|---|---|
| スケーリング速度 | 分単位 | 秒単位 |
| ゼロスケール | 可能 | 不可 |

## ネットワーク・セキュリティ

### SG vs NACL
| | SG | NACL |
|---|---|---|
| 適用範囲 | インスタンス単位 | サブネット単位 |
| ステート | ステートフル | ステートレス |
| 拒否ルール | なし | あり |

- 特定IPを拒否したい → **NACL**

### WAF配置可能な場所
- CloudFront・ALB・API Gateway・AppSync ✅
- NLB・EC2直接 ❌

## その他サービスメモ

- **Amplify** = フロントエンド＋バックエンドをまるごと簡単構築
- **Elastic Beanstalk** = バックエンドアプリの簡単デプロイ
- **Lightsail** = 小規模・月額固定のVPS
- **Pinpoint** = マーケティングメール・SMS・プッシュ通知の配信
- **SES** = システム自動通知メール（注文確認等）
- **SR-IOV（拡張ネットワーキング）** = 物理NICに直接アクセスして高速化
- **CloudFront TTL** = キャッシュ保持時間・即削除はInvalidationを使う
