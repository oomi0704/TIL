# AWS SAA メモ（2026-04-14）

## Global Accelerator
- **役割**: 静的 Anycast IP で受信 → AWS グローバルネットワーク経由で TCP/UDP をバックエンド（ALB / NLB / EC2 / EIP 等）へ。ヘルスチェック・フェイルオーバー。
- **ALB vs NLB**: どちらもエンドポイントにできる。**HTTP(S)・L7（パス/ホスト等）** → ALB。**TCP/UDP・非 HTTP・低レイテンシ** → NLB。
- **CloudFront との違い**: CloudFront は HTTP(S) の CDN（キャッシュ中心）。GA は TCP/UDP、経路最適化寄り。

## Amazon DLM（Data Lifecycle Manager）
- EBS **スナップショット / AMI** の作成・保持・削除を**ポリシーで自動化**。バックアップ運用とコスト管理。
- S3 Lifecycle（オブジェクトの階層・削除）とは別物。

## CloudTrail / CloudWatch / エージェント / Logs
| サービス | 範囲・できること |
|----------|------------------|
| **CloudTrail** | **API 操作の監査**（誰が・いつ・何を）。コンプライアンス・変更追跡。 |
| **CloudWatch** | **メトリクス**の可視化・**アラーム**・ダッシュボード。 |
| **CloudWatch エージェント** | EC2/オンプレから **ログ・カスタム/システムメトリクス** を Logs / メトリクスへ送信。 |
| **CloudWatch Logs** | **ログ**の保存・検索（Insights）・フィルターでメトリクス化・アラーム・他サービスへ転送。 |

## Kinesis Data Streams vs Amazon Data Firehose
| | Data Streams | Data Firehose |
|---|--------------|---------------|
| 位置づけ | ストリームに保持し、**自分でコンシューマ**（Lambda 等）で読む | **バッファしながら** S3 / Redshift / OpenSearch 等へ**マネージド配達** |
| レイテンシ | 低レイテンシ・リアルタイム向き | ニアリアルタイム（バッファ） |
| 試験の目安 | 複数読み手・カスタム処理 | ログ/分析基盤への取り込み |

## Streams + Firehose を組み合わせる理由
- **Streams** = 共有の受け皿（複数コンシューマ・保持・再読込みしやすい）。
- **Firehose** = データレイク等への配達専門（圧縮・バッファ・宛先ロード）。
- 例: `プロデューサー → Streams` 分岐で **Lambda（即時処理）** と **Firehose → S3（分析・長期保管）** を同じイベントで両立。
