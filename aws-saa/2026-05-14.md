w# TIL - 2026-05-14 AWS SAA 学習まとめ

## 1. ENI（Elastic Network Interface）
- EC2にアタッチする仮想NIC
- プライベートIP・ElasticIP・MACアドレス・セキュリティグループを持つ
- 特定のAZに紐づく
- 複数のENIを1つのインスタンスにアタッチ可能

---

## 2. EFS vs FSx
- **EFS**：Linux専用・NFSプロトコル・汎用共有ストレージ・自動拡張
- **FSx for Windows**：Windows・SMBプロトコル・AD連携
- **FSx for Lustre**：機械学習・HPC・S3と直接連携

---

## 3. Egress-Only Internet Gateway
- IPv6専用の送信専用ゲートウェイ
- アウトバウンド（内部→外部）のみ許可
- インバウンド（外部→内部）は拒否
- IPv4のNAT Gatewayに相当するIPv6版

---

## 4. Direct Connect vs Site-to-Site VPN

| 項目 | Direct Connect | Site-to-Site VPN |
|---|---|---|
| 回線 | 専用線 | インターネット |
| 速度 | 高速・安定 | 変動あり |
| 暗号化 | なし（別途必要） | IPsecで自動暗号化 |
| 導入速度 | 数週間〜数ヶ月 | 数時間〜数日 |
| コスト | 高い | 安い |

- 両者ともオンプレ↔AWSをつなぐトンネルとして機能は同じ
- セキュアかつ高速→Direct Connect + VPN併用が最強

---

## 5. その他のトンネル系サービス
- **Client VPN**：個人PC↔AWS
- **VPN CloudHub**：複数拠点をAWS経由で接続
- **VPC Peering**：VPC同士の接続
- **Transit Gateway**：複数VPCをハブ型で一括接続
- **PrivateLink**：サービス単位のプライベート接続

---

## 6. Cost Anomaly Detection
- 機械学習でコスト異常を自動検知
- メール・SNSで通知
- AWSサービス別・アカウント別・タグ別で監視可能
- **AWS Budgets**との違い：Budgetsは手動閾値設定、こちらはML自動検知

---

## 7. Kinesis Data Streams vs Firehose

| 項目 | Data Streams | Firehose |
|---|---|---|
| リアルタイム性 | ミリ秒 | 準リアルタイム（数秒〜数分） |
| 管理 | 手動シャード管理 | フルマネージド |
| データ保持 | 最大365日 | なし |
| 宛先 | 自由 | S3・Redshift・OpenSearch等固定 |

- リアルタイム・複数コンシューマー・再処理 → **Data Streams**
- S3/Redshiftへ自動配信・管理不要 → **Firehose**

---

## 8. DynamoDB・DAX・ElastiCache

### DAX
- DynamoDB専用インメモリキャッシュ
- ミリ秒→マイクロ秒に高速化
- 読み取り集中ワークロードに有効
- 書き込み集中には不向き

### ElastiCache
| | Redis | Memcached |
|---|---|---|
| 用途 | セッション管理・ランキング | シンプルなキャッシュ |
| 永続化 | あり | なし |
| マルチAZ | あり | なし |
| スケール | 垂直 | 水平 |

- **DynamoDBのキャッシュ → DAX**
- **RDS・その他のキャッシュ → ElastiCache**

---

## 9. DB選択まとめ

```
SQLが必要？
├── Yes → リレーショナルDB
│    ├── コスト重視 → RDS
│    ├── 高性能・高可用性 → Aurora
│    └── 変動トラフィック → Aurora Serverless
└── No → NoSQL
     ├── 大規模Key-Value → DynamoDB
     ├── ドキュメント → DocumentDB
     ├── グラフ → Neptune
     ├── 時系列 → Timestream
     └── 台帳 → QLDB
```

### Aurora vs Aurora Serverless
- **Aurora**：常時起動・固定キャパシティ・安定トラフィック向け
- **Aurora Serverless v2**：自動スケール・トラフィック予測不能・開発環境向け

---

## 10. EBS ボリュームタイプ

| 種類 | 用途 |
|---|---|
| gp3 | 汎用・デフォルト・コスパ最強 |
| io2 | 高IOPS・ミッションクリティカル・Oracle/SAP |
| io2 Block Express | 超大規模DB・最大256,000 IOPS |
| st1 | ビッグデータ・ログ処理 |
| sc1 | コールドデータ・最安 |

- **HDD系（st1・sc1）はブートボリュームに使用不可**
- **io3は存在しない**

### マルチアタッチ
- io1・io2のみ対応
- 同一AZ内・最大16インスタンス
- クラスター対応ファイルシステムが必要

### RAID
- **RAID0**：パフォーマンス最大化・冗長性なし
- **RAID1**：ミラーリング・冗長性あり

---

## 11. S3転送高速化

| サービス | 用途 |
|---|---|
| Transfer Acceleration | S3へのアップロード高速化・エッジロケーション経由 |
| CloudFront | S3からのダウンロード高速化 |
| Global Accelerator | EC2・ALBへのアクセス高速化・固定Anycast IP |

---

## 12. Direct Connectのメリット
- 安定した通信品質（専用線）
- 高速・低遅延（1Gbps〜100Gbps）
- セキュリティ高（インターネット非経由）
- 大量転送時のコスト削減
- ハイブリッドクラウド構成に最適

---

## 13. ECS vs EKS

| 項目 | ECS | EKS |
|---|---|---|
| 基盤 | AWS独自 | Kubernetes |
| 学習コスト | 低い | 高い |
| 移植性 | AWSに依存 | マルチクラウド対応 |
| コスト | 安い | 高い |

- **Kubernetesを使いたい・マルチクラウド → EKS**
- **AWSシンプルに使いたい → ECS**
- **Podレベルの権限管理 → EKS + IRSA**
- **サーバー管理不要 → Fargate**

---

## 14. SAA vs SAP レベル差

| 項目 | SAA | SAP |
|---|---|---|
| 難易度 | ★★★☆☆ | ★★★★★ |
| 問題数 | 65問 | 75問 |
| 試験時間 | 130分 | 180分 |
| 出題範囲 | 広く浅く | 広く深く |

- SAA：「何を使うか知っている」
- SAP：「なぜその構成にするか説明できる」
- まずSAAに集中→実務経験→SAP挑戦が王道
