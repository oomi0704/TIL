## 1. AWS Global Accelerator

- エッジロケーションからAWSバックボーン経由で最適なリージョンへ届けるサービス
- **Anycast IP**：2つの静的グローバルIPが付与される
- 自動フェイルオーバー・ヘルスチェック・重み付けルーティングが可能

### CloudFrontとの使い分け
| | Global Accelerator | CloudFront |
|--|--|--|
| 用途 | TCP/UDPの低遅延通信 | HTTPコンテンツ配信（キャッシュ） |
| キャッシュ | なし | あり |
| 向いてる用途 | ゲーム・VoIP・API・IoT | 静的コンテンツ・動画配信 |
| IPアドレス | 固定（Anycast） | 変動 |

### IoTデバイスはGlobal Acceleratorを使う
- MQTTなどTCP/UDPプロトコルを使うため
- リアルタイム性・低遅延が重要なため
- センサーデータはキャッシュ不要のため

## 2. CloudFront vs Route 53

- **CloudFront**：CDN（コンテンツ配信・キャッシュ）
- **Route 53**：DNS（名前解決・ルーティング）

```
ユーザーが「example.com」にアクセス
① Route 53 → 名前解決（道案内）
② CloudFront → キャッシュ配信（荷物を届ける）
```

### Route 53のルーティングポリシー
| ポリシー | 内容 |
|---------|------|
| シンプル | 単純に1つのIPを返す |
| レイテンシーベース | 最も遅延が少ないリージョンに誘導 |
| 地理的（Geolocation） | ユーザーの地域でルーティング |
| 加重（Weighted） | 割合でトラフィックを分散 |
| フェイルオーバー | 障害時にスタンバイへ切り替え |

## 3. マルチリージョンでのCloudFront活用

- CloudFront自体がグローバル（エッジロケーション400箇所以上）
- **オリジンフェイルオーバー**：障害時に別リージョンへ自動切り替え
- **S3マルチリージョンアクセスポイント**と組み合わせで低遅延配信
- 動的コンテンツのマルチリージョンはRoute 53と組み合わせる

## 4. Aurora vs RDS

| 項目 | RDS | Aurora |
|------|-----|--------|
| 対応DB | MySQL/PostgreSQL/Oracle/SQL Server/MariaDB | MySQL互換/PostgreSQL互換のみ |
| ストレージ | 単一AZのEBS | 3AZ×2コピー=6コピー自動複製 |
| レプリカ数 | 最大5台 | 最大15台 |
| フェイルオーバー | 約1〜2分 | 約30秒以内 |
| パフォーマンス | 標準 | MySQL比最大5倍・PostgreSQL比最大3倍 |

### Aurora独自機能
- **Aurora Serverless**：アクセス量に応じて自動スケール
- **Aurora Global Database**：複数リージョンへレプリケーション（1秒以内）
- **Backtrack**：データを過去の時点に巻き戻せる

### クロスリージョンリードレプリカ
- RDS・Aurora両方が持っている
- AuroraはGlobal Databaseを使うとより高速（遅延1秒以内・最大5リージョン）

## 5. STS AssumeRole

- 一時的な認証情報を取得して別のIAMロールの権限を借りる仕組み
- 発行される情報：AccessKeyId / SecretAccessKey / SessionToken（有効期限15分〜12時間）

### 主なユースケース
| ユースケース | 内容 |
|-------------|------|
| クロスアカウントアクセス | 別アカウントのリソースにアクセス |
| EC2からのアクセス | アクセスキーをコードに埋め込まなくて済む |
| 外部IDプロバイダー連携 | Google・Active Directoryなど |

### AssumeRoleの種類
| API | 用途 |
|-----|------|
| AssumeRole | AWSユーザー/ロールが別ロールを引き受ける |
| AssumeRoleWithWebIdentity | Google・Cognitoなど外部IDで引き受ける |
| AssumeRoleWithSAML | Active DirectoryなどSAML認証で引き受ける |

## 6. IAM ロール・ポリシーの使い分け

### ポリシーの種類
| 種類 | 内容 |
|------|------|
| アイデンティティベースポリシー | ユーザー/グループ/ロールに付与 |
| リソースベースポリシー | S3バケットポリシーなどリソース側に設定 |
| 信頼ポリシー | ロールを誰が引き受けられるか定義 |
| Permission Boundary | 権限の上限を設定 |
| SCP | AWS Organizations全体に適用する上限 |

### 権限の評価順序
1. 明示的なDeny → 即拒否
2. SCPで許可？
3. リソースベースポリシーで許可？
4. Permission Boundaryで許可？
5. アイデンティティポリシーで許可？
6. デフォルトDeny

## 7. EBS（Elastic Block Store）

- EC2にアタッチする仮想ハードディスク
- **主な用途**：OS・アプリ・ライブラリ・設定ファイルの格納
- 単一AZ内でのみアタッチ可能

### EBSの種類
| 種類 | 用途 |
|------|------|
| gp3/gp2 | 汎用SSD・一般的なワークロード |
| io1/io2 | 高性能SSD・DB・高IOPSが必要な場合 |
| st1 | スループット最適化HDD・ビッグデータ |
| sc1 | コールドHDD・アクセス頻度が低いデータ |

### S3・RDSが必要な理由
- EBSは基本的に1つのEC2にしかアタッチできない
- S3：大容量・複数EC2からのアクセス・低コスト
- RDS：同時書き込み・整合性・自動管理

## 10. WAF / CloudFront / Geo Match 使い分け

| シナリオ | 答え |
|---------|------|
| 映像配信×特定の国に制限 | **CloudFront Geo Restriction** |
| 国別制限＋他のセキュリティルール | WAF + Geo Match |
| SQLインジェクション防御 | WAF |
| DDoS対策 | CloudFront + WAF + Shield |

### ポイント
- Geo MatchはWAFの機能の一つ（単独では使えない）
- 単純な国別制限 → CloudFront Geo Restriction（追加コストなし）
- 複雑な条件 → WAF Geo Match
