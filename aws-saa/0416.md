# AWS SAA メモ（2026-04-16）

会話で確認した論点（Firehose と SQS、クリックストリーム、CloudFront、AI サービス群、SageMaker）を試験向けに整理。

---

## Firehose と SQS の違い

### Kinesis Data Firehose

- 目的: ストリーミングデータを S3 / Redshift / OpenSearch などへ配信する。
- 特徴: サーバレス、バッファリングしてまとめて配信、変換（Lambda）や圧縮も可能。
- 向いている用途: ログやイベントの大量取り込み、分析基盤への連携。
- 注意点: 「個々のメッセージをアプリで厳密に再処理する」用途は得意ではない。

### SQS

- 目的: システム間の非同期メッセージキュー。
- 特徴: コンシューマが取りに行くモデル、再試行、DLQ、疎結合化に強い。
- 種類:
  - Standard: 少なくとも 1 回配信（重複の可能性あり）。
  - FIFO: 順序保証 + 重複排除。
- 向いている用途: ワーカー処理、業務トランザクションの非同期化。

### 「取りこぼし」の整理

- Firehose は配信失敗時のリトライ機能を持つ。
- 配信先障害や変換失敗時は、エラーデータの退避設定（例: S3 バックアップ）を設計していないとロスに見える。
- 試験では「大量ログの配送」は Firehose、「再処理制御が重要な業務キュー」は SQS が基本。

## クリックストリーム（Clickstream）

- 定義: ユーザーの操作履歴（ページ閲覧、クリック、検索、カート追加など）の時系列ログ。
- 典型構成:
  - 収集: Kinesis（Data Streams / Firehose）
  - 保存: S3
  - 分析: Athena / Redshift / EMR
- 問題文キーワード: 「大量・連続・ほぼリアルタイム・行動分析」。

---

## CloudFront の CDN 用途

- ユーザーに近いエッジロケーションから配信し、低レイテンシ化する。
- キャッシュによりオリジン（S3 / ALB / EC2）負荷を下げる。
- HTTPS、WAF、Shield、署名付き URL/Cookie などで安全な配信を実現できる。
- 試験キーワード: 「グローバル配信」「高速化」「キャッシュ」「オリジン負荷軽減」。

## AI サービスの使い分け（頻出）

- Amazon Rekognition:
  - 画像/動画解析（顔・物体・不適切コンテンツ検出など）。
- Amazon Textract:
  - OCR + 構造抽出（フォーム項目、表データ）。
- Amazon Comprehend:
  - 一般テキストの NLP（感情分析、エンティティ抽出など）。
- Amazon Comprehend Medical:
  - 医療テキスト専用 NLP（病名、薬剤、症状などの抽出）。

覚え方:

- 画像/動画 -> Rekognition
- 書類 OCR -> Textract
- 一般文章 NLP -> Comprehend
- 医療文章 NLP -> Comprehend Medical

## SageMaker

- 役割: 機械学習モデルの作成・学習・デプロイを一気通貫で実施する基盤。
- 選ぶ場面: 独自モデルを構築したい、学習と推論 API 公開まで管理したい。
- 他サービスとの違い:
  - Rekognition / Textract / Comprehend は「既製 AI を使う」。
  - SageMaker は「独自モデルを作る」。
- 問題文キーワード: 「カスタムモデル」「トレーニング」「ハイパーパラメータ調整」「推論エンドポイント」。

---

## 今日の試験向けショートメモ

- 大量ログ配送なら Firehose、再試行制御が必要な業務連携なら SQS。
- クリックストリームは「収集 -> 蓄積 -> 分析」のパイプラインで考える。
- CloudFront は高速化だけでなくオリジン保護・セキュリティ強化にも効く。
- AI 系は「入力データの種類（画像/文書/一般文/医療文）」で切り分ける。
- 「独自 ML モデル」が見えたら SageMaker を第一候補にする。
