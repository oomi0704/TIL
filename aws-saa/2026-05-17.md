## 1. S3 ストレージクラスの違い

### Standard-IA（Infrequent Access）
- アクセス頻度が低いが、必要時はすぐ取得したいデータ向け
- 取得速度：ミリ秒（即時）
- 最小保存期間：30日
- 可用性：99.9%
- 取得料金が発生

### Glacier Instant Retrieval
- めったにアクセスしないが即時取得が必要なアーカイブデータ向け
- 取得速度：ミリ秒（即時）
- 最小保存期間：**90日**
- 取得料金は高め

### Intelligent-Tiering
- アクセスパターンが**予測できない**データ向け
- 監視・自動化料金が発生するが、取得料金は**無料**
- 内部Tier構造で自動移動（Frequent → Infrequent → Archive Instant → Archive → Deep Archive）

### 試験のポイント
| シナリオ | 答え |
|---------|------|
| アクセスパターンが不明・予測できない | Intelligent-Tiering |
| 取得料金を払いたくない | Intelligent-Tiering |
| 低頻度（月1回）でわかっている | Standard-IA |
| 非常に低頻度（年1〜2回）かつ即時取得必要 | Glacier Instant Retrieval |

## 2. EBS ボリュームタイプ（IOPS）

### gp2 vs gp3
| 項目 | gp2 | gp3 |
|------|-----|-----|
| IOPSとサイズの関係 | 連動（3IOPS/GB） | 独立して設定可能 |
| コスト | 普通 | 20%安い（推奨） |
| 最大IOPS | 16,000 | 16,000 |

### io1 vs io2
| 項目 | io1 | io2 |
|------|-----|-----|
| 耐久性 | 99.8〜99.9% | **99.999%（5ナイン）** |
| IOPS/GB比率 | 最大50:1 | 最大500:1 |

### 試験のポイント
- コスパ重視の一般用途 → `gp3`
- 64,000 IOPSが必要 → `io1` または `io2`
- Multi-Attach（複数EC2から同時マウント） → `io1` / `io2` のみ
- 最高の耐久性（5ナイン） → `io2`

## 3. セキュリティグループ vs ネットワークACL

| 項目 | セキュリティグループ | ネットワークACL |
|------|------------------|---------------|
| 適用レベル | インスタンス単位 | サブネット単位 |
| ステートフル/レス | ステートフル | ステートレス |
| ルールの種類 | 許可のみ | 許可・拒否両方 |
| ルールの評価順 | 全ルールを評価 | 番号順（小さい番号優先） |

### 重要：ステートレスの意味
- ACLはレスポンスのアウトバウンドも**別途明示的に許可**が必要
- エフェメラルポート（1024-65535）の許可を忘れずに

### VPC全体への制限
- ネットワークACLはサブネット単位のため、VPC全体に直接適用するものはない
- 方法①：全サブネットに同じACLを紐付ける
- 方法②：**AWS Network Firewall**（VPC全体・高度なフィルタリング）

### 試験のポイント
- 特定IPを拒否したい → ネットワークACL（SGは拒否不可）
- インスタンス単位で制御 → セキュリティグループ
- VPC全体で制御 → Network Firewall

## 4. VPCエンドポイントと PrivateLink

### ゲートウェイ型エンドポイント
- 対応サービス：**S3・DynamoDBのみ**
- 料金：**無料**
- 仕組み：ルートテーブルにルートを追加
- オンプレからの利用：不可

### インターフェース型エンドポイント（= PrivateLink）
- 対応サービス：ほぼ全てのAWSサービス
- 料金：有料（時間課金＋データ転送料）
- 仕組み：サブネットにENI（プライベートIP）を作成
- オンプレからの利用：**可能**（Direct Connect/VPN経由）

### 試験のポイント
| シナリオ | 答え |
|---------|------|
| S3へインターネットを使わず接続・コスト最小 | ゲートウェイ型（無料） |
| CloudWatchへプライベート接続 | インターフェース型（PrivateLink） |
| 自社SaaSを他社VPCへ安全に公開 | PrivateLink + NLB |
| オンプレからAWSサービスへプライベート接続 | インターフェース型（PrivateLink） |

## 5. API Gateway vs Lambda Function URL

### Lambda Function URL
- 単一LambdaへHTTPSエンドポイントを直接付与
- 料金：Lambda料金のみ（安い）
- 認証：IAM or なし
- throttling・WAF・カスタムドメイン：なし

### API Gateway
- 複数LambdaをまとめてAPI管理
- throttling・WAF・Cognito・カスタムドメイン・キャッシュ・ステージ管理：あり
- 料金：高め

### 試験のポイント
| シナリオ | 答え |
|---------|------|
| 外部コールバック・Webhookを受け取るだけ | Lambda Function URL |
| コスト最小でLambdaを公開 | Lambda Function URL |
| WAF・throttling・Cognitoが必要 | API Gateway |
| 複数LambdaをルーティングしてAPI化 | API Gateway |

> **キーワード判断のコツ**
> 「Webhook・コールバック + 管理機能の記述なし」→ Function URL
> 「WAF・throttling・Cognito・カスタムドメイン」→ API Gateway
