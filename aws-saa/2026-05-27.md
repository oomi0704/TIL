# TIL - 2026-05-27 AWS SAA 学習メモ

## API Gateway vs Lambda Function URL
- **API Gateway**: 認証・WAF・カスタムドメイン・ステージ管理が必要な本番API向け
- **Lambda Function URL**: シンプル・コスト最小・Webhook等の簡易エンドポイント向け
- RESTful APIが出たらAPI Gateway一択
- API GatewayにはREST APIとHTTP APIがあり、HTTP APIの方が安くシンプル

## ベクトルストレージ
- テキスト・画像を数値配列（ベクトル）に変換して保存し、意味の近さで検索するDB
- RAG構成（生成AI）で必須の技術
- 関連サービス：OpenSearch Service、Aurora PostgreSQL（pgvector）、Bedrock Knowledge Bases

## Amazon DocumentDB
- MongoDBと互換性のあるフルマネージドドキュメントDB
- JSONドキュメント形式・スキーマレス
- DynamoDBとの使い分け：複雑なクエリ→DocumentDB、シンプルなKVS→DynamoDB

## スタンドアロン
- 単独稼働・冗長化なしの構成
- 試験では「改善してください」のパターンで登場することが多い

## Polly / Lex / Transcribe
- **Polly**: テキスト→音声（読み上げ）
- **Lex**: 会話を理解してチャットボットを作る
- **Transcribe**: 音声→テキスト（文字起こし）
- 組み合わせ：Transcribe（音声→テキスト）→ Lex（理解）→ Polly（読み上げ）

## Amazon Fraud Detector
- 機械学習で不正行為（決済・偽アカウント）を自動検出
- GuardDuty（AWSインフラへの脅威）と混同しないこと

## Route53 ルーティングポリシー
- **シンプル**: 単一エンドポイント
- **フェイルオーバー**: 障害時自動切り替え（ヘルスチェック必須）
- **加重**: 割合で振り分け（A/Bテスト・段階移行）
- **レイテンシー**: 最速リージョンに自動転送
- **Geolocation**: 国・地域でルール固定
- **Geoproximity**: 距離＋バイアスで柔軟調整
- **トラフィックフロー**: 複数ポリシーをGUIで組み合わせ・再利用

## VPCエンドポイント ゲートウェイ型
- S3とDynamoDBのみ対応・無料
- プライベートサブネットからインターネット経由せずアクセス可能
- NATゲートウェイ経由はコストがかかるため非推奨

## AWS Backup
- 複数AWSサービスのバックアップを一元管理
- 対象：EC2・RDS・Aurora・DynamoDB・EFS・FSx・S3等
- **ボールトロック（WORM）**: バックアップの削除・変更を禁止
- クロスリージョン・クロスアカウントコピーに対応

## Babelfish for Aurora PostgreSQL
- SQL ServerのT-SQLをそのままAurora PostgreSQLで動かせる互換レイヤー
- アプリの改修コストを最小化してSQL ServerからAWSに移行できる

## Fargate
- 試験では「サーバーレス」として扱う
- ECS/EKSの起動タイプとして使用
- Lambdaとの使い分け：15分以上・コンテナ→Fargate、短時間・イベント駆動→Lambda

## Amazon EKS
- AWSのマネージドKubernetes
- ノードタイプ：セルフマネージド・マネージドノードグループ・Fargate
- ECSとの使い分け：Kubernetes必須・移植性重視→EKS、シンプル→ECS

## CloudFront vs ElastiCache Redis vs Route53レイテンシー
- **CloudFront**: コンテンツをエッジでキャッシュして返す（CDN）
- **ElastiCache Redis**: DBクエリ結果をメモリでキャッシュ（DB高速化）
- **Route53レイテンシー**: 最速リージョンにDNSで誘導するだけ（キャッシュなし）
- 3つは競合ではなく組み合わせて使うもの

## IAM グループ vs ロール
- **グループ**: IAMユーザーをまとめる箱・ポリシーを一括適用
- **ロール**: 一時的な権限の貸し出し券・AWSサービスや人に付与
- グループにアタッチできるのはポリシーのみ（ロールはアタッチ不可）
- ロールはグループにアタッチできない

## SCP vs IAMポリシー
- **SCP**: 権限の上限・ガードレール・rootユーザーも制限可能
- **IAMポリシー**: 実際の権限を付与する
- 両方の許可が必要（どちらか一方でも拒否→操作不可）
- SCPはPrincipalが使えない・JSON形式は同じ

## RDS マルチAZ vs リードレプリカ
- **マルチAZ**: 可用性・障害対策・同期レプリケーション・自動フェイルオーバー・読み取り不可
- **リードレプリカ**: 読み取り性能向上・非同期・手動昇格・最大5台・別リージョン可
- 組み合わせて同時使用可能

## クロスリージョンリードレプリカ
- 対応：MySQL・PostgreSQL・MariaDB・Aurora
- 非対応：SQL Server（スナップショットコピーで代替）
- AuroraグローバルDBの方が高速フェイルオーバー（1分以内・遅延1秒以下）

## Redshift
- 列指向ストレージ・MPP（大規模並列処理）でOLAP（分析）に特化
- **Redshift Spectrum**: S3のデータをロードせず直接クエリ
- AthenaはRedshift不要・単独でS3をクエリ
- SpectrumはRedshiftのデータとS3をJOINできる

## AWS Snow Family
- **Snowcone**: 8TB・2.1kg・現場・遠隔地
- **Snowball Edge Storage**: 80TB・大量データ転送
- **Snowball Edge Compute**: 42TB・エッジでEC2・Lambda実行可能
- **Snowmobile**: 100PB・トラック・データセンター丸ごと移行
- 目安：10TB以上でSnowball検討

## DataSync
- オンプレ・AWS間・AWS同士・他クラウド→AWSのデータ転送・同期
- 対応：S3・EFS・FSx・オンプレNFS/SMB
- 差分転送・スケジュール・整合性チェックに対応
- 似たサービス：S3レプリケーション・Storage Gateway・Transfer Family・Snow Family

## Storage Gateway
- **File Gateway**: ファイルをS3に保存（NFS・SMB）
- **Volume Gateway キャッシュ型**: S3メイン・オンプレはキャッシュ
- **Volume Gateway 蓄積型**: オンプレメイン・S3はバックアップ
- **Tape Gateway**: テープバックアップをS3・Glacierに

## VPCピアリングとセキュリティグループID参照
- 同一VPC内・同一リージョンピアリング → SGID参照可能
- クロスリージョンピアリング・クロスアカウント → SGID参照不可・CIDRで代替

## S3 クロスリージョンレプリケーション（CRR/SRR）
- S3固有のサービス・バージョニング有効化が必須
- CRR：別リージョン複製（DR・低レイテンシ）
- SRR：同一リージョン複製（ログ集約・環境コピー）
- 既存オブジェクトはS3 Batch Operationsで対応

## DMS（Database Migration Service）
- 使う場面：異種DB間移行・ダウンタイム最小化・継続的レプリケーション
- 異種DB移行はSCT（スキーマ変換）+ DMSをセットで使用
- 同種DB・AWS内移行はスナップショット・mysqldump等でDMS不要
