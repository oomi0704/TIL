## パイロットライト（Pilot Light）とは

- **災害復旧（DR）の方式のひとつ**。本番と同じ構成の**ごく一部だけ**を、多くは**別リージョン**で常時・小さく動かしておく考え方。
- ガスコンロの**常備灯（パイロット）**の比喩。本番ほど大きくは燃やさないが、いつでも本番規模に「着火」できる状態にしておく。

### 常時あるもの

- **データの複製**（例：別リージョンの DB レプリカ、S3 のクロスリージョンレプリケーションなど）。
- **必要最小限のコンピュート**（フル台数ではなく最小構成や、停止可能な待機など）。

### 障害時にすること

- 待機側で **Auto Scaling や追加インスタンス** でスケールアウトする。
- **Route 53** などで DNS／トラフィックの向き先を切り替える。
- 本番に近い構成まで**短時間で拡張**する。

## DR方式の位置づけ（ざっくり比較）

| 方式 | 常時コスト | 復旧の速さ（目安） |
|------|------------|-------------------|
| バックアップ＆リストア | 低い | 遅め |
| **パイロットライト** | 中程度 | その中間 |
| ウォームスタンバイ | やや高い | 速め |
| アクティブ／アクティブ | 高い | 最速に近い |

## 試験向けメモ

- **RTO / RPO**：バックアップのみより改善しやすいが、アクティブ／アクティブほど即時ではない、という整理が出やすい。
- 選択肢で「**最小限を常時＋障害時に拡張**」「データはレプリカ等で待機」と書かれていれば、パイロットライトの説明と一致するかを確認する。
- **コストと復旧速度のトレードオフ**の問題では、パイロットライトは「中コスト・中程度のRTO」に置く。

## Storage Gateway: キャッシュ型とストア型の違い

- ここでの比較は、Volume Gateway の **Cached Volumes（キャッシュ型）** と **Stored Volumes（ストア型）**。

### 結論

- **キャッシュ型（Cached Volumes）**：データ本体は主に **AWS（S3）**。オンプレ側は頻繁に使うデータをキャッシュ。
- **ストア型（Stored Volumes）**：データ本体は主に **オンプレミス**。AWS 側はスナップショットなどバックアップ用途。

### 比較ポイント

- **主データの置き場所**
  - キャッシュ型：AWS
  - ストア型：オンプレ
- **オンプレ必要容量**
  - キャッシュ型：少なくて済む
  - ストア型：多めに必要
- **主な目的**
  - キャッシュ型：オンプレ容量を節約しつつクラウド中心で運用
  - ストア型：既存オンプレ運用を維持しつつ AWS へバックアップ/DR
- **アクセス特性**
  - キャッシュ型：ホットデータはローカルで高速、コールドデータは AWS から取得
  - ストア型：通常アクセスはローカル中心で低遅延

### 試験向けメモ

- 「**オンプレの容量不足**」「**データはAWSに寄せたい**」なら **キャッシュ型**。
- 「**データはオンプレ保持が必須**」「**クラウドへバックアップしたい**」なら **ストア型**。

## DynamoDB Streams

- DynamoDB テーブルのアイテムに対する **変更イベント（INSERT / MODIFY / REMOVE）** を時系列で取得できる機能。
- 変更をトリガーにして、**Lambda で非同期処理**を実行する構成が定番。

### 何に使うか

- **監査ログ**や変更履歴の取得
- **集計テーブルの更新**（マテリアライズドビュー的な運用）
- 別システムへの **イベント連携**（通知、検索インデックス更新、キャッシュ更新など）

### レコードに含める情報（StreamViewType）

- `KEYS_ONLY`：キーのみ
- `NEW_IMAGE`：更新後の全属性
- `OLD_IMAGE`：更新前の全属性
- `NEW_AND_OLD_IMAGES`：更新前後の両方

### 制約・注意点

- ストリームレコードの保持期間は **最大24時間**。
- 主な用途は **near real-time のイベント処理**であり、長期保管用途ではない。
- Lambda と組み合わせる場合は、**再試行**や**冪等性**を考慮して設計する。

### 試験向けメモ

- 「DynamoDB の変更をきっかけに自動処理」「コードを疎結合に連携」とあれば、まず **DynamoDB Streams + Lambda** を検討。
- 「更新前後の値が必要」なら `NEW_AND_OLD_IMAGES` を選ぶ。
