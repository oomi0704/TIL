## 1. Apache Flink

- リアルタイムのストリームデータ処理フレームワーク
- AWSでは **Amazon Managed Service for Apache Flink** として提供
- Kinesis Data Streams や MSK（Kafka）と組み合わせて使う
- サーバーレスで インフラ管理不要

---

## 2. Firehose vs Apache Flink

| | Firehose | Apache Flink |
|---|---|---|
| 目的 | データの**配送** | データの**処理・分析** |
| リアルタイム性 | 準リアルタイム（数十秒〜） | 真のリアルタイム（ミリ秒） |
| コード | 不要 | 必要 |

- 両方を同時に使う構成が頻出
  - Firehose → S3（保存用）
  - Flink → リアルタイム分析

---

## 3. Amazon MSK（Managed Streaming for Apache Kafka）

- フルマネージドの Kafka サービス
- **既存のKafkaをAWSに移行したい場合**に選ぶ
- Kinesisとの違い
  - メッセージサイズ：MSKは最大1GB、Kinesisは最大1MB
  - 保持期間：MSKは無制限、Kinesisは最大365日
- **MSK Serverless**：キャパシティ管理不要で自動スケーリング

---

## 4. バックアップの種類と保存期間

| 種類 | サービス | 保存期間 |
|---|---|---|
| PITR | DynamoDB / RDS / Aurora | 最大**35日** |
| 自動スナップショット | RDS / Redshift | 最大**35日** |
| 手動スナップショット | EBS / RDS / Redshift | **無制限** |
| AWS Backup | 各種サービス | **自由に設定可能** |

- **35日以上前に戻したい** → PITRでは不可、手動スナップショットを使う
- **AWS Backup** = PITRやスナップショットを束ねる管理ツール

---

## 5. データ移行・同期サービスの使い分け

| サービス | 用途 |
|---|---|
| **DataSync** | ファイルの移行・定期同期 |
| **Storage Gateway** | オンプレとAWSの恒久的なハイブリッド接続 |
| **Snowball** | ペタバイト級データの物理輸送 |
| **DMS** | データベースの移行・同期 |
| **Transfer Family** | SFTP/FTPでS3/EFSへ転送 |
| **S3 Transfer Acceleration** | S3へのアップロードを高速化 |

---

## 6. PrivateLink・エンドポイントの使い分け

| サービス | 用途 |
|---|---|
| **Gateway Endpoint** | S3・DynamoDBへのプライベート接続（**無料**） |
| **Interface Endpoint** | ほぼ全AWSサービスへのプライベート接続（有料） |
| **PrivateLink** | サービスを他VPCに安全に公開 |
| **VPC Peering** | 少数VPC間の接続（1対1） |
| **Transit Gateway** | 大規模・多数VPCの一元接続 |

- PrivateLinkで外部APIは叩けない → **NAT Gateway**を使う
- 一部のSaaS（Datadog, Snowflake等）はPrivateLink対応あり

## 7. S3 Standard-IA vs S3 Glacier Instant

| | S3 Standard-IA | S3 Glacier Instant |
|---|---|---|
| アクセス頻度 | 月1回程度 | 四半期に1回程度 |
| 取り出し速度 | ミリ秒 | ミリ秒 |
| 最低保存期間 | 30日 | 90日 |

- **「コンプライアンス監査」** というキーワード → 頻度不明でも **Standard-IA**
  - 理由：定期的な義務 = アクセス頻度が自然と高くなる

## 8. AWS ECR

- Dockerイメージを保存・管理するフルマネージドなコンテナレジストリ
- GitHubのDockerイメージ版のイメージ
- CI/CDパイプライン：GitHub → CodeBuild → **ECR** → ECS/EKS
- 主な機能
  - **イメージスキャン**：脆弱性検出
  - **ライフサイクルポリシー**：古いイメージを自動削除

---

## 9. EC2 Fleet

- 複数のインスタンスタイプ・購入オプションを組み合わせて一括調達
- 配分戦略
  - **lowest-price**：最安のプールから調達
  - **diversified**：複数プールに分散
  - **capacity-optimized**：中断リスクを最小化（試験頻出）

---

## 10. AWS Batch vs Lambda

| | Lambda | AWS Batch |
|---|---|---|
| 最大実行時間 | **15分** | **無制限** |
| サーバーレス | 完全 | Fargate使用時のみ |
| 用途 | 軽量・イベント駆動 | 重い・長時間バッチ処理 |

- AWS Batch + **Fargate** = サーバーレス
- AWS Batch + **EC2** = サーバーレスではない

---

## 11. Amazon EKS 頻出ポイント

- ECS vs EKS
  - 既存KubernetesをAWSに移行 → **EKS**
  - AWS新規構築・シンプル → **ECS**
- 複数Podでストレージ共有 → **EFS**
- PodにAWSリソースへの権限付与 → **IRSA**
- Kubernetes Ingress はSAA試験にはほぼ出ない

---

## 12. CloudFront vs Redis（ElastiCache）

| | CloudFront | Redis |
|---|---|---|
| キャッシュ対象 | 静的・動的コンテンツ | DBクエリ・セッション |
| 場所 | エッジロケーション（世界中） | VPC内（DBの手前） |
| 目的 | 配信の高速化 | DBの負荷軽減 |

- 2段階キャッシュ構成が頻出
  ```
  ユーザー → CloudFront → EC2/ECS → Redis → RDS
  ```
